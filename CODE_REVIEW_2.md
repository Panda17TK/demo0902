## A. 今回の堅牢化で新たに生じた懸念（★）

### 🟠★ 固定ステップで実効シミュレーションが120Hzになり CPU が約2倍
`core/constants.js` の `FIXED_DT = 1/120`、`main.js:155-214` のアキュムレータ。
60fps 表示なら `dt≈0.0167`、`FIXED_DT≈0.0083` で、**1描画フレームあたり更新パイプライン
（combat+ai+spawner+items+fx＋`buildMobGrid`）が約2回実行**される。
旧実装は実質60Hz（可変1ステップ）だったため、**CPU負荷がおよそ倍増**。
せっかく空間分割で稼いだ分を固定ステップが食う構図。
→ 方向性: `FIXED_DT=1/60` で十分（弾18px/step < 32pxタイルで安全）。120Hz が要るのは
高速移動体のみなので、ステップ刻みを 1/60 にし、blink 等だけ内部サブステップ化。

### 🟡★ 30fps 端末でシミュレーションが減速（時間落とし）
`main.js:158` `MAX_STEPS=5`。重い端末で `dt` が大きいと、必要ステップ数が上限を超え
`accumulator=0` で**時間を捨てる＝ゲームがスローになる**。描画と乖離。
→ 方向性: `MAX_STEPS` を上げる/`FIXED_DT` を粗くする、または可変ステップへの劣化フォールバック。

### 🟡★ 固定ステップなのにレンダリング補間がない（120Hz sim / 60Hz 描画）
描画は `state` の生座標を読む（`renderer.js`）。sim が 120Hz で描画が 60Hz だと、
2ステップ進んだ位置をそのまま描くため、**理論上はスムーズだが、`accumulator` の端数分
（0〜FIXED_DT）が毎フレームの表示位相をわずかに揺らす**。今は2倍刻みなので目立たないが、
`FIXED_DT` を粗くすると顕在化する。
→ 方向性: 前ステップ位置を保持し `alpha=accumulator/FIXED_DT` で線形補間。

### 🟠★ 近傍グリッドを毎ステップ `new Map()`＋文字列キーで再構築（GC圧）
`spatial.js:12-26` の `buildMobGrid` は毎回 `new Map()`、各 mob で `cx+','+cy` の**文字列キー生成**、
セル毎に `new Array`。これを**毎フレーム2回**（固定ステップ2回）。敵28体で
Map1＋文字列28＋配列十数個/ステップ。**O(n²)は消えたが新たな per-frame アロケーション**を作った。
→ 方向性: 数値キー（`cx*STRIDE+cy`）化、グリッドの使い回し（clear して再利用）、
または mob 配列を bucket にバケットソートして添字で参照。

### 🟡★ グリッドのクエリ半径が mob 自身の大きさを考慮せず取りこぼし得る
`spatial.js:30-40` の `forNearby` はクエリ中心±radius のセルのみ走査。**mob の半幅
（ボス w=46→23px）を加味しない**。`CELL=48`、弾ヒットのクエリ半径は `combat.js` で 24px。
中心が別セルにある大型 mob が、境界付近で**当たり判定から漏れる**可能性。
（現状 makeMobFromKey の最大 w は 46 で CELL 48 に近く、リスクは小さいが原理的欠陥）
→ 方向性: クエリ半径に「想定最大 mob 半径」を加える、もしくはセルを最大 mob より大きく取る。

### 🔴★ ビームのグリッド照会が線形走査より約58倍遅い（最優先の実害）
`combat.js`（ビーム）は `maxL=700 / step=6 ≒117ステップ`、各ステップで `forNearby(...,24,...)`。
1ステップで約2セル × セル内 mob（密集時 ~7体）を走査するため、**117×2×7 ≒ 1638 回**の判定。
旧来の「敵28体を1回だけ線形走査＝28回」と比べ**約58倍**の比較回数になり、しかも毎回
`Map.get(文字列キー)` のハッシュ往復が乗る。**空間分割が完全に逆効果**になっている箇所。
（独立した第三者監査でも CRITICAL 判定）
→ 方向性: ビームは「線分 vs mob 中心の垂線距離」で解析的に判定し、グリッドは
「線分の AABB に重なる mob を1回だけ集める」用途に限定。最低でも最初のステップで
集めた mob を Set にキャッシュして再照会しない。

### 🟡★ グリッドの陳腐化（AI 分離が「移動前」位置で計算）
`main.js:161-169` の実行順は `buildMobGrid → combat → tiles → spawner → ai → items → fx`。
グリッドはステップ先頭で構築されるが、**mob が動くのは `updateAI` 内**（`moveAndCollide`、
さらに blink で二度動く）。そのため `updateAI` の分離処理（`ai.js` の `forNearby`）は
**移動前の古い位置**で近傍を判定する＝「いま重なっている敵」を正しく分離しきれず団子が残る。
combat の弾/斬撃/爆発は ai より前に走るのでグリッドは新鮮（実害は分離に限定）。
→ 方向性: 分離は ai 内で「その場の座標」で判定する（グリッドは粗いふるいに留め、最終判定は
平方距離で実座標比較）か、ai の直前にグリッドを1回更新。

### 🟢★ dev-editor の編集反映が武器/敵で不統一
`render/dev-editor.js:142` 武器編集は「次のリスタートから反映」。一方
敵ステータス/攻撃は `makeMobFromKey` が `CONFIG.enemies` を毎回読むため**次スポーンから即時**。
プレイヤー武器は `state.player.weapons` が起動時コピーのため**ラン中は固定**。一貫性がない。
→ 方向性: ステータス系（dmg/fireRate）は発射時に CONFIG を参照すれば武器もライブ化可能。

## B. 前回からの積み残し（未対応）

### 🟠 combat.js / renderer.js の肥大化は未分割
`combat.js` は 464行（むしろ微増）、`renderer.js` は 368行。FX レジストリ化は済んだが、
combat の「移動/近接/射撃/弾更新/敵弾/斬撃/爆発」一体は残存。
→ 方向性: `weapons.js`/`projectiles.js`/`melee.js` 分割。

### 🟡 アイテムが if 連鎖＋ハードコード（データ駆動化されず）
`items.js:8-17` 取得の if 連鎖、`renderer.js` 描画の if 連鎖、バフ値（×2・15s 等）が
`items.js:14-16` にハードコード（CONFIG 外＝dev-editor で調整不可）。
→ 方向性: `CONFIG.items` テーブル（効果・色・グリフ・確率・バフ倍率/時間）に集約。

### 🟡 アイテム取得が全アイテム線形走査
`items.js:5` は毎ステップ全 `state.items` を走査（半径 AABB）。敵ドロップで増えると線形コスト。
固定ステップで2倍に。
→ 方向性: アイテムにもグリッド/簡易空間索引、または取得判定を間引き。

### 🟡 マップが単一ハードコード文字列（複数面なし）
`state/map.js` 依然 `RAW` 1枚。さらに**初期スポーンだけ旧 `makeZombie/makeSpitter`（ai.js 経由）**を
使い（`map.js:49-50`）、波中のデータ駆動スポーンと**生成経路が二系統**。
→ 方向性: マップを JSON 化＋初期スポーンも `makeMobFromKey` に統一。

### 🟠 認証が弱い（uid = "demo-"+sessionId）
`ControllerServlet.java:20` のまま。なりすまし/レート制限なしは未対応。
→ 方向性: 実験用途なら README に明記。本運用なら実認証＋トークン。

### 🟡 DAO のフルスキャン / 非アトミック書き込み
`FileScoreDAO.listTop` は毎回全行読み（`:56`）、`FileGameSaveDAO.list` は毎回全列挙。
`Files.write`（`FileScoreDAO:47`/`FileGameSaveDAO:51`）は temp＋ATOMIC_MOVE でなく、
クラッシュ時に半端書き込みの恐れ。
→ 方向性: 上位スコアのメモリキャッシュ、原子的書き込み。

### 🟢 未使用 MySQL コネクタ同梱（2.4MB）
`WEB-INF/lib/mysql-connector-j-8.1.0.jar` は未使用のまま WAR 同梱。
→ 方向性: 除去 or 実際に使う DAO を追加。

### 🟢 DAO がスタティック singleton（テストが副作用依存）
`DAOFactory` の static は不変。`-Darpg.persistence=memory` 前提のテストのまま。
→ 方向性: DI 化。

### 🟢 サーバ側ロギング不在 / `escape` の制御文字未対応
`ApiGameStateV2Servlet.escape`（`:139`相当）は今も `\` と `"` のみ。スロット一覧の
`updatedAt` 等に制御文字が混じると JSON 破損の余地（slot 名はサニタイズ済みなので実害は限定的）。

## 優先度つき次アクション（提案）

1. **🔴 ビームのグリッド照会を是正**（線形 or 解析判定）— 唯一の明確な実害（約58倍）。最優先。
2. **🟠 固定ステップを 1/60 に**（CPU 倍増の解消）＋必要なら blink だけ内部サブステップ。

   `CELL ≥ 最大mob径 + 2×最大クエリ半径` の制約コメントを `constants.js` に明記。
4. **🟠 combat.js の分割**（weapons/projectiles/melee）。
5. **🟡 アイテムのデータ駆動化**（`CONFIG.items`）＋取得の空間索引、AI 分離の陳腐化是正。
6. **🟡 マップの JSON 化**＋初期スポーンの `makeMobFromKey` 統一（生成経路の一本化）。
7. **🟢 描画グラデーションのキャッシュ**、未使用 MySQL jar 除去、認証の位置づけを README 明記。
